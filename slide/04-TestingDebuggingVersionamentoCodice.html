Tipologie di test

  • Unit Test

Unit Test

  • Unit Test (Test Unitari)
    • Verificano la singola unità di codice (una funzione o un metodo o un'intera classe)
    • Deve essere isolato, non dipendere da servizi esterni
    • Mock: si utilizzano per simulare comportamenti di componenti esteni
    • TaMaSy: Testare se la funzione calcola_scadenza() restituisce la data corretta
  • Integration Test
  • System Test

Perché testare

  • Prevenzione dei bug
    • Trovare l'errore mentre si scrive il codice costa meno che farlo trovare all'utente
  • Modifiche future sicure
    • Posso cambiare il codice interno sapendo che, se i test passano, non ho rotto le funzionalità esistenti
  • Documentazione viva
    • I test spiegano come deve comportarsi il codice meglio di mille commenti

Caratteristiche di un buon test

  • Principi F.I.R.S.T.
    • Fast - esecuzione in pochi secondi
    • Independent - non deve dipendere dal risultati di un altro test
    • Repeatable - deve dare lo stesso risultato sempre in ogni ambiente
    • Self-validating - risultato binario PASS (verde) o FAIL (rosso)
    • Thorough/Timely (Completo/Tempestivo) - copre i casi limite e viene scritto insieme al codice

Ciclo TDD

  • Fase 1 - RED
    • Scrivi un test per una funzione che non esiste ancora. Il test fallisce
  • Fase 2 - GREEN
    • Scrivi il minimo codice necessario per far passare il test
  • Fase 3 - REFACTOR
    • Migliora il codice appena scritto (pulizia, performance) mantenendo il test verde

Motto: "Prima scrivi il test, poi scrivi il codice"

Unit Testing in Django

  • Tool Integrato: Django usa il modulo unittest di Python ma lo potenzia
  • Database di Test: Ogni volta che lanciamo python manage.py test, Django crea un database temporaneo vuoto (non tocca i dati reali!)
  • Il Client di Django: Simula un browser per testare le View (richieste GET/POST) senza aprire il browser
  • Comando: python manage.py test

Le asserzioni comuni

  • assertEqual(a, b): a è uguale a b?
  • assertNotEqual(a, b): a è diverso da b?
  • assertTrue(x) / assertFalse(x): la condizione è vera/falsa?
  • assertContains(response, "testo"): La pagina HTML contiene questa parola?
  • assertRaises(Error): Il codice lancia l'errore corretto in caso di dati sbagliati?

Anatomia di un test

  1. Set Up: Preparazione dei dati (creazione utenti, task di prova)
  2. Execution: Esecuzione della logica da testare (chiamata a una funzione o visita a una URL)
  3. Validation (Assert): Controllo del risultato
  4. Tear Down: Pulizia (gestita automaticamente da Django)

Integration Test

Oltre l'isolamento

  • E' la fase di testing in cui i moduli vengono combinati
  • Obiettivo
    • Verificare che le interfacce e la comunicazione tra i diversi pezzi del sistema funzionino correttamente
  • Focus
    • la view deve leggere correttamente dal database
    • la sicurezza è garantita per le operazioni disponibili?
    • i dati passano bene da back-end a front-end e viceversa?
    • ...

Integration Test con Django

  • In un'applicazione web come Django, la distinzione tra Unit e Integration è sottile perché quasi tutto interagisce con il Database
  • Verifichiamo
    • Integrità del Database: Un task salvato tramite una form viene scritto correttamente nella tabella
    • Flusso di Navigazione: Dopo il login, l'utente viene reindirizzato correttamente alla sua dashboard
    • Middleware e Sicurezza: I filtri di sicurezza di Django (CSRF, Authentication) proteggono le rotte sensibili
  • Si usa ancora django.test.TestCase ma si usa il client

System Test

  • Unit Test
  • Integration Test
  • System Test
    • Verificano l'intera applicazione come se fossimo l'utente finale

L'App nel suo insieme

  • È la fase in cui il sistema software viene testato nella sua interezza. Non testiamo più i singoli componenti, ma il prodotto finito
  • Approccio "Black Box": Il tester non guarda il codice interno (le classi Python o Java), ma interagisce con l'applicazione come se fosse l'utente finale, verificando ingressi e uscite
  • Validare se il sistema soddisfa i Requisiti Funzionali (cosa deve fare l'app) e i Requisiti Non Funzionali (prestazioni, sicurezza, usabilità)

Ambiente

  • Viene eseguito in un ambiente di "Staging"
    • Una copia esatta di quello che sarà l'ambiente di produzione reale (il web server finale)
  • I test in produzione sono vietati
    • Porre la massima restistenza quando viene chiesto di farlo

TaMaSy System Test

Test non funzionali

  • Verifiche
    • Usabilità: È facile per un nuovo utente capire come creare un task?
    • Sicurezza: Se provo ad accedere alla dashboard senza login, il sistema mi blocca sempre?
    • Carico: Cosa succede se 50 utenti salvano un task nello stesso istante?
  • Strumenti Professionali
    • In questa fase si usano spesso tool di automazione browser come Selenium o Playwright, che "guidano" il browser al posto dell'uomo

Debugging

  • L'Arte del Detective
    • Uso del Debugger: Il programmatore non "indovina", ma "osserva"
    • Breakpoints: Punti di interruzione in cui il programma si ferma per permetterci di ispezionare e modificare i valori in memoria
    • Watch Expressions: Tenere d'occhio una variabile specifica mentre il codice scorre riga per riga
    • Analisi Errori: Differenza tra errori di sintassi (il codice non parte) e bug logici (il codice parte ma fa cose sbagliate)

Debug in VS Code

  • Il Breakpoint (Punto di Interruzione):
    • Si inserisce cliccando a sinistra del numero di riga (apparirà un pallino rosso)
    • Indica a VS Code di mettere in pausa il programma esattamente prima di eseguire quella riga
  • Come Avviare:
    • Premere F5 o andare sull'icona "Run and Debug" nella barra laterale
    • Scegliere "Python File" (o "Django" se stiamo testando unàapplicazione Django)
  • Perché non usare il print()?
    • Il debug non sporca il codice, permette di vedere tutte le variabili insieme e di modificare lo stato del programma "al volo"

Navigazione nel Debug

  • Quando il codice si ferma su un breakpoint, appare una barra degli strumenti in alto. Ecco come muoversi
    • Continue (F5) Riprende l'esecuzione
    • Step Over (F10) Esegue la riga corrente
    • Step Into (F11) Se nella riga c'è una chiamata di funzione entra nella funzione
    • Step Out (Shift+F11) Esce dalla funzione corrente
    • Restart (Ctrl+Shift+F5) Riavvia il programma da capo
    • Stop (Shift+F5) Chiude la sessione di debug

Version control

  • Version Control con Git (Il "Tasto Annulla" dei programmatori)
    • Concetti base: Repository (la cartella del progetto), Commit (il salvataggio dello stato), Push/Pull (invio e ricezione dati dal server)
    • GitHub: Non è solo un backup, è il "social network" del codice e lo strumento di hosting per il portfolio degli studenti
    • TaMaSy: Carichiamo TaMaSy su GitHub: la vostra prima impronta nel mondo professionale

Git inizializzazione

  • Configurazioni iniziali
    • git config --global user.name "Tuo Nome": Imposta il nome dell'autore
    • git config --global user.email "email@esempio.it": Imposta l'email

Git repository

  • Ottenere un repository
    • git init: Crea un nuovo repository locale nella cartella corrente. Apparirà una cartella nascosta .git
    • git clone <url>: Copia un repository esistente da GitHub sul tuo PC

Git ciclo di lavoro

  • git status Il comando più importante. Ti dice quali file sono stati modificati e cosa sta succedendo
  • git add <file>: Aggiunge un file alla Staging Area (lo prepara per il salvataggio). Usate git add . per aggiungere tutto
  • git commit -m "Descrizione modifica": Crea un punto di ripristino (snapshot). La descrizione deve essere chiara
  • Ordine delle operazioni: Modifica file ➔ git add ➔ git commit

Sincronizzazione con il repository di origine

  • Relazione con i repository remoti
    • git remote add origin <url>: Collega il tuo repository locale a quello remoto
    • git push: Carica i tuoi commit locali sul server remoto (GitHub)
    • git pull: Scarica le ultime modifiche fatte dai tuoi colleghi e le unisce al tuo codice
    • git fetch: Controlla se ci sono novità sul server senza scaricarle ancora nel tuo codice

Branching (Lavorare in parallelo)

  • Creare numerose linee di sviluppo
    • git branch <nome-ramo>: Crea un nuovo ramo (es: feature-notifiche)
    • git checkout <nome-ramo>: Si sposta sul ramo indicato (o git switch)
    • git merge <nome-ramo>: Unisce le modifiche del ramo indicato nel ramo in cui ti trovi (solitamente main)

Ispezione e ripristino

  • Cosa fare quando qualcosa va storto o vogliamo vedere la storia
    • git log: Mostra la cronologia di tutti i commit (chi ha fatto cosa e quando)
    • git diff: Mostra le differenze esatte tra il codice attuale e l'ultimo salvataggio
    • git checkout -- <file>: Annulla le modifiche non salvate a un file, riportandolo all'ultimo commit
    • git reset --soft HEAD~1: Torna indietro di un commit ma mantiene le modifiche nei file (per correggere un errore nel messaggio di commit

Workflow Collaborativo (Teamwork)

  • Branching (Rami): il ramo main è il prodotto finito. Ogni sviluppatore lavora su un ramo separato (es. feature-notifiche) per non rompere il lavoro degli altri
  • Merge & Pull Request (PR): Il processo di richiesta di unione del proprio codice nel ramo principale
  • Gestione Conflitti: Cosa succede se due persone modificano la stessa riga?
  • Code Review: Leggere il codice degli altri sviluppatori per suggerire miglioramenti prima che venga accettato

Documentazione

  • Scrivere per gli esseri umani
    • Docstring: Commenti speciali dentro il codice per spiegare cosa fa una funzione
    • README.md: Il file più importante di ogni progetto GitHub. Deve spiegare cos'è TaMaSy, come si installa e come si usa
    • TaMaSy Slide: "Se un altro programmatore legge il vostro codice, deve capire come farlo girare in 5 minuti grazie al README."
